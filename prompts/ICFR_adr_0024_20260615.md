# ADR-0024: git push 승인 원칙 (2-tier)

- **상태**: 채택 (Accepted)
- **일자**: 2026-06-10
- **관련 ADR**: ADR-0021 (협업 룰)
- **결정자**: 마스터(CFO/창업자), Claude(기술 자문)

## Context

ICFR 프로젝트는 마스터 1인 의사결정 + Claude 기술 자문 + Regina(프론트엔드) 협업 구조다. push 시점마다 발생하는 두 가지 비효율·위험이 있다.

- 모든 push를 마스터가 일일이 확인하면 가벼운 문서 수정·오타 픽스까지 병목이 걸려 작업 속도가 떨어진다.
- 반대로 무분별하게 자동 push 하면 모델/스키마 변경, 데이터 마이그레이션 등 되돌리기 어려운 변경이 검토 없이 main에 올라가 감사 추적성(ICFR 제도 취지)과 운영 안정성을 해칠 수 있다.

따라서 "무엇을 자동으로 밀고, 무엇을 마스터 승인 후 밀지"의 명확한 기준이 필요하다.

## Decision

push 대상을 위험도에 따라 2단계로 분류한다.

### Tier 1 — 자동 push (Claude 판단으로 즉시 push)

다음에 해당하면 마스터 승인 없이 commit·push 한다.

- 문서·ADR·명세서(prompts/) 추가 및 수정
- 명백한 버그 픽스 (동작·계약 변경 없음)
- 주석·로그·포맷·네이밍 등 동작에 영향 없는 변경
- AI(Claude)가 추천한 결정대로 진행해도 되는 것으로 합의된 가벼운 변경

### Tier 2 — 마스터 승인 push (마스터가 직접 push)

다음에 해당하면 Claude는 변경을 준비·제안만 하고, 실제 push는 마스터가 수행한다.

- 신규 기능 추가
- 데이터 모델 / 스키마 / API 계약 변경
- 데이터 마이그레이션을 동반하는 변경
- 인프라(서버·DB·Docker·CI 등) 구성 변경
- 시드 데이터 또는 운영 데이터에 영향을 주는 변경

### 판단 축

분류가 모호하면 아래 3축으로 판단하고, 하나라도 높으면 Tier 2로 올린다.

1. **되돌리기 난이도** — revert 한 번으로 원복되는가, 데이터 손실/마이그레이션 롤백이 필요한가
2. **데이터·스키마 영향** — 기존 데이터·계약·감사 추적에 영향을 주는가
3. **AI 추천 신뢰도** — Claude 단독 판단으로 충분히 안전한가, 마스터의 사업·제도 판단이 필요한가

## Consequences

- **즉시 적용 사례**: 2026-06-10 작업4 remediation·search enrichment 명세서 자동 push (`4d9c119`)가 본 원칙 Tier 1의 첫 적용 사례다.
- 가벼운 변경의 작업 속도가 올라가고, 위험 변경은 마스터 검토 게이트를 반드시 통과한다.
- 분류 경계 사례가 누적되면 본 ADR의 예시 목록을 갱신한다.
- Regina의 작업은 branch → PR → TrustBuilder 리뷰/머지 흐름을 따르므로 본 원칙은 main 직접 push 판단에 적용된다 (ADR-0021 참조).
