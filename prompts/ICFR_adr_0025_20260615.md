# ADR-0025: ICFR 플랫폼 근간 아키텍처 — Baseline + Overlay + 멀티테넌시

- **상태**: 채택 (Accepted)
- **일자**: 2026-06-15
- **결정자**: 마스터(CFO/창업자)
- **관련 ADR**: ADR-0020(제로 추상화), ADR-0022(영업자료 축적), ADR-0023(데이터 복구)
- **성격**: 시스템 전체를 좌우하는 근간 결정. 모든 후속 모듈은 이 구조 위에 배치된다.

---

## Context

기존 시스템은 사이냅소프트 RCM 템플릿(93개 통제)을 `controls` 테이블에 직접 적재한 **단일 회사 고정 스키마**였다. 그러나 제품 목표는 **모든 회사에 적용 가능한, 법령 완전 대응 기준선(Basic-perfect)에서 출발해 회사별로 무한 변형 가능한 ICFR 관리 플랫폼**이다.

이 목표와 현 구조 사이에는 근본 충돌이 있다 — 통제 항목·속성·코드 체계가 코드에 고정되어 있어 회사별 커스터마이징이 불가능하다. 고정 코딩이 더 쌓이기 전, 지금이 뼈대를 바로잡을 유일한 시점이다.

핵심 통찰: **제도(법)가 강제하는 골격은 불변이고, 그 안을 채우는 내용만 가변이다.** 기업이 국가(법)를 넘을 수 없으므로, 평가 흐름의 큰 틀은 모든 회사 공통이다.

---

## Decision

### 1. 2층 구조 — 불변 골격 / 가변 내용

**불변 골격 (제도가 강제, 회사가 변경 불가)**
- 평가 흐름: 설계평가 → 운영평가 → 미비점 → 개선 → 보고
- 이 단계의 존재와 순서는 ICFR 제도의 법적 요구사항이며 회사가 바꿀 수 없다.
- 시스템은 이 골격을 코드로 보장한다 (워크플로 엔진/룰 엔진 불필요 — 골격이 고정이므로).

**가변 내용 (회사가 결정)**
- 통제 항목, 속성, 코드 체계, 평가 기준, 담당/결재 대상.
- baseline에서 비채택/추가/변형으로 회사별 맞춤.

### 2. Baseline + Overlay 모델

**Baseline (Basic-perfect)** — 법령·제도에 완전 대응하는 표준 RCM/통제 집합. 시스템 자산이며 회사와 무관하다. 법령 개정 시 이 계층만 갱신한다.

**Overlay (회사별 결정 레이어)** — baseline에 대한 회사의 결정만 분리 저장한다:
- **비채택**: baseline 통제를 플래그로 제외 (삭제 아님 — 원본 보존)
- **변형**: baseline 통제를 참조하되 변경 필드만 오버라이드
- **추가**: 회사 고유 신규 통제 레코드

**결합 방식: 참조 + 차분(diff)**. 회사 인스턴스는 baseline을 **참조**하고 변경분만 overlay에 저장한다. 미변경 통제는 baseline을 그대로 따르므로, **baseline 교체 시 미변경분이 자동 갱신**된다 (법령 개정 대응). baseline을 복제·편집하는 방식(법 개정이 회사마다 수동 반영)은 채택하지 않는다.

이 분리는 세 가지를 동시에 만족한다:
- 법령 개정 시 baseline만 교체, overlay 유지
- 감사 추적: "회사가 표준에서 무엇을 왜 뺐는가"가 overlay에 기록됨
- AI 자동설정 비전: AI가 생성하는 것이 곧 overlay (기업 인터뷰 → 비채택/추가/변형 결정)

### 3. 메타데이터 동적 속성 — 핵심 고정 + 확장 필드

- **제도 공통 속성은 컬럼으로 고정**한다: 통제명, 주기, 핵심통제 여부, 어서션, 예방/적발, 자동/수동 등 — 제도가 규정하는 거의 모든 회사 공통 속성.
- **회사 고유 추가 속성만 확장 필드(JSON/동적)**로 둔다.
- 완전 EAV(모든 속성을 정의 데이터화)는 채택하지 않는다 — owner_name 정렬·검색 같은 기능마다 쿼리·검증·성능 복잡도가 폭증하며, ICFR은 제도가 속성을 상당 부분 규정하므로 실익이 적다.

### 4. 멀티테넌시 — 전 테이블 tenant_id 상시, 단일 코드베이스

- **모든 테이블에 `tenant_id`를 항상 포함**한다.
- **SaaS 배포**: 공유 DB에 다수 tenant 격리 (애플리케이션 레벨 tenant_id 필터).
- **온프레/전용 배포**: 고객 자체 서버에 단독 구축. 구조는 SaaS와 동일하며, 단지 그 인스턴스에 **tenant가 1개만 존재**할 뿐이다.
- 별도 DB 구조나 코드 분기를 만들지 않는다. 온프레 = "tenant 1개짜리 SaaS". **단일 코드베이스, 배포 설정만 상이.**

### 5. 평가 골격은 코드로 보장

설계평가→운영평가→미비점→개선→보고 흐름은 고정이므로, 회사별 가변 워크플로 엔진을 도입하지 않는다. 단계 내부의 파라미터(평가 기준·담당)만 가변. 복잡도를 통제하면서 제도 준수를 보장한다.

## Consequences

### Phase 1 재배치 필요 (직언)

현재 RCM·Test·Remediation·Evidence는 baseline/overlay/tenant 개념 없이 단일 회사 고정으로 구현되어 있다. 이 뼈대로 가려면 **3축(baseline / overlay / tenant) 위로 재배치**해야 한다. 코딩이 적게 쌓인 지금이 정확한 타이밍이며, 미루면 회사 증가 시 비용이 제곱으로 커진다 (retrofit 폭탄).

재배치 범위(후속 명세에서 상세화):
- 전 테이블 `tenant_id` 추가
- `controls`를 baseline 통제와 회사 overlay로 분리
- 회사 인스턴스의 통제 조회 = baseline 참조 + overlay diff 병합 로직
- 기존 사이냅소프트 93개 통제 데이터를 baseline 또는 첫 tenant overlay로 정리

### 마이그레이션 전략

기존 고정 스키마 → 가변 뼈대 전환은 데이터 보존이 핵심이다 (ADR-0023). 마이그레이션은 단계적으로, 각 단계 검증 후 진행한다. 한 번에 전체를 엎지 않는다.

### AI 자동설정 비전과의 접점

기업 인터뷰 → AI 분석 → 자동 RCM 생성은, 본 구조에서 **AI가 overlay를 생성**하는 것으로 자연히 구현된다. baseline은 고정 자산, AI는 회사별 결정 레이어만 만든다. 본 아키텍처가 그 비전의 토대다.

### 후속 작업

- baseline/overlay 데이터 모델 상세 설계 (별도 명세)
- tenant_id 전면 도입 마이그레이션 (별도 명세)
- Phase 1 4개 모듈 3축 재배치 순서 결정
- 그 후 Phase 2 모듈(scoping·euc·iuc·schedule·notification·report) 진입

## 미결정 (후속 ADR/명세에서)

- baseline 버전 관리 (법령 개정 이력)
- tenant 생성/온보딩 플로우
